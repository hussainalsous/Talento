الملخص التنفيذي

كشف تدقيق أمني شامل لنظام مصادقة TALENTO عن خمس ثغرات أمنية متميزة — تتراوح بين حرجة (غياب تام لتقييد المعدل) وعالية الخطورة (رموز لا تنتهي صلاحيتها). تمّت معالجة جميعها بتطبيق محدود يمسّ ثلاثة ملفات فقط، دون أي تبعيات جديدة، مع إعادة استخدام بنية Redis الموجودة أصلًا لخدمة التخزين المؤقت.

المشكلات المكتشفة قبل التقوية

حرج

لا يوجد تقييد لمعدل تسجيل الدخول

نقطة النهاية POST /api/v1/auth/login خالية تمامًا من أي حماية لتقييد الطلبات. يمكن للمهاجم إرسال محاولات لا محدودة لتخمين كلمة المرور دون أي عائق.

حرج

لا توجد حماية من هجمات القوة الغاشمة

لم يُسجَّل أي عداد للمحاولات الفاشلة على الإطلاق. حسابات المدير (admin) معرّضة للهجوم عبر نفس النقطة غير المحمية كحسابات المستخدمين العاديين.

عالٍ

خطر تعداد المستخدمين

الحسابات الموقوفة كانت تُعيد رسالة خطأ مختلفة ("تم تعطيل حسابك") مقارنةً بحسابات غير موجودة، مما يتيح للمهاجم معرفة ما إذا كان بريد إلكتروني معيّن مسجّلًا في النظام.

عالٍ

ثغرة هجوم التوقيت

عند استخدام بريد إلكتروني غير مسجّل، كان PHP يتخطى عملية bcrypt بالكامل عبر Short-Circuit Evaluation، مما يجعل الاستجابة أسرع بكثير — وهو فارق زمني قابل للقياس يكشف عن وجود البريد الإلكتروني في النظام.

عالٍ

رموز Sanctum لا تنتهي صلاحيتها أبدًا

كان الإعداد 'expiration' => null في ملف config/sanctum.php يجعل الرموز صالحة للأبد. أي رمز مسروق يمنح وصولًا دائمًا للحساب.

ما تم تطبيقه

تقييد معدل الطلبات

Laravel RateLimiter بمفتاح مركّب login:{email}:{ip}، 5 محاولات، حظر لمدة 15 دقيقة، مدعوم بـ Redis الحالي.

الحماية من القوة الغاشمة

عداد المحاولات يزيد عند كل فشل، ويُصفَّر عند تسجيل الدخول الناجح. يُعاد HTTP 429 عند تجاوز الحد.

منع تعداد المستخدمين

جميع مسارات الفشل — بريد غير موجود، كلمة مرور خاطئة، حساب موقوف — تُعيد رسالة خطأ موحدة تمامًا. السبب الحقيقي يُسجَّل داخليًا فقط.

التخفيف من هجمات التوقيت

يُستدعى Hash::check() دائمًا بغض النظر عن وجود المستخدم. تجزئة وهمية بتكلفة 12 تضمن تنفيذ bcrypt الكامل في كل الحالات.

انتهاء صلاحية الرمز

رموز Sanctum تنتهي بعد 7 أيام. لا حاجة لهجرة قاعدة بيانات — Sanctum يستخدم created_at + expiration في كل طلب.

التسجيل الأمني

كل محاولة فاشلة وكل حدث حظر يُسجَّل بـ Log::warning() مع البريد، IP، السبب، والتوقيت. كلمات المرور لا تُسجَّل أبدًا.

الحالة الأمنية النهائية

✅
تمت معالجة جميع الثغرات الأمنية الخمس الحرجة وعالية الخطورة. يطبّق نظام المصادقة الآن المجموعة الكاملة من الدفاعات المعيارية المطلوبة لأي API في بيئة إنتاجية: حماية متعددة الأبعاد من القوة الغاشمة، ومنع تعداد المستخدمين، وتكافؤ التوقيت، ورسائل خطأ موحدة، وانتهاء صلاحية الرموز بعد 7 أيام، وتسجيل أمني منظّم — وكل ذلك باستخدام مكونات Laravel الأساسية دون أي تبعيات جديدة.

تقييد تسجيل الدخول

يُستخدم Laravel RateLimiter الأصيل لفرض حدّ على المحاولات في نقطة نهاية تسجيل الدخول. بما أن CACHE_STORE=redis مضبوط مسبقًا، يخزّن المحدّد عداداته تلقائيًا في Redis دون أي إعداد إضافي.

إعدادات محدد المعدل

صيغة المفتاحlogin:{email_بأحرف_صغيرة}:{IP_العميل} الحد الأقصى للمحاولات5 محاولات فاشلة مدة الحظر900 ثانية (15 دقيقة) الإعادة عند النجاحنعم — RateLimiter::clear() عند تسجيل الدخول الناجح الخادم الخلفيRedis (عبر CACHE_STORE=redis — لا إعداد إضافي) استجابة HTTP عند الحظر429 Too Many Requests النطاقلكل زوج بريد إلكتروني + IP (هجمات IP الموزعة لا تتجاوز حماية البريد)

تصميم المفتاح: مركّب من البريد الإلكتروني + IP

استخدام مفتاح مركّب من البريد الإلكتروني وعنوان IP يوفر توازنًا مثاليًا: IP واحد يحاول على حسابات متعددة يُرصد عبر البُعد IP، وحساب واحد يُهاجَم من IPs متعددة يُرصد عبر البُعد البريدي. المزيج يمنع قفل المستخدم الشرعي بسبب محاولات مستخدم آخر من نفس الشبكة.

app/Services/AuthService.php PHP 
// ثوابت مستوى الفئة — مصدر وحيد للإعدادات الأمنية
private const MAX_ATTEMPTS  = 5;
private const DECAY_SECONDS = 900; // 15 دقيقة

public function login(string $email, string $password, string $ip): array
{
    $throttleKey = 'login:' . Str::lower($email) . ':' . $ip;

    // فحص: هل تجاوز هذا المفتاح الحد الأقصى؟
    if (RateLimiter::tooManyAttempts($throttleKey, self::MAX_ATTEMPTS)) {
        Log::warning('security.login_rate_limited', [...]);
        abort(429, 'Too many login attempts. Please try again later.');
    }

    // ... التحقق من بيانات الاعتماد ...

    // عند الفشل: زيادة العداد
    RateLimiter::hit($throttleKey, self::DECAY_SECONDS);

    // عند النجاح: مسح العداد
    RateLimiter::clear($throttleKey);
}
ℹ️
لماذا لم نستخدم middleware throttle: على مستوى المسار؟ الـ middleware يستخدم IP فقط مفتاحًا ويعدّ جميع أنواع الطلبات. التطبيق على مستوى الخدمة يستخدم المفتاح المركّب email+IP ويزيد العداد فقط عند الفشل — مما يجعله أكثر دقة وأقل احتمالًا لقفل المستخدمين الشرعيين.

الحماية من هجمات القوة الغاشمة

سيناريو الهجوم

في هجوم حشو بيانات الاعتماد (Credential Stuffing) أو رش كلمات المرور (Password Spray)، يجرب المهاجم مئات أو آلاف كلمات المرور الشائعة على عناوين بريد إلكتروني معروفة. بدون حماية، كل محاولة تستغرق فقط وقت فحص bcrypt (حوالي 150-200 مللي ثانية عند التكلفة 12)، مما يعني أن أداة آلية تستطيع تنفيذ آلاف المحاولات في الدقيقة.

❌ السلوك السابق

  • لا يوجد عداد للمحاولات من أي نوع
  • كل طلب معالَج باستقلالية — الفشل السابق لا يؤثر على التالي
  • يمكن للمهاجم إرسال محاولات لا محدودة على مدار الساعة
  • حسابات المدير معرّضة لنفس الهجوم الغير محدود
  • لا رؤية أمنية — لا سجلات، لا تنبيهات

✅ السلوك الجديد

  • عداد المحاولات مُخزَّن في Redis لكل مفتاح بريد+IP
  • العداد يزيد عند كل محاولة فاشلة
  • بعد 5 إخفاقات: حظر 15 دقيقة مع HTTP 429
  • تسجيل الدخول الناجح يمسح العداد فورًا
  • كل إخفاق مُسجَّل مع البريد والـIP والسبب والوقت

الفوائد الأمنية

تقييد سرعة الهجوم

5 محاولات كل 15 دقيقة = 20 محاولة/ساعة لكل زوج بريد+IP. كلمة مرور من 6 أحرف صغيرة بها 300 مليون تركيبة — بهذا المعدل يستغرق الاستنفاد أكثر من 1700 سنة.

الحفاظ على وصول المستخدمين الشرعيين

العداد يُمسح عند النجاح. مستخدم أخطأ كلمة المرور 4 مرات ثم أدخلها صحيحة، يُفتح حسابه فورًا — لا تدخل يدوي من المدير.

مقاومة الهجمات الموزعة

المفتاح المركّب بريد+IP يعني أن المهاجم الذي يستخدم شبكة botnet بـIP مختلف لكل طلب لا يزال يُعدّ كل IP منفصلًا. 100 IP يمنح 500 محاولة فقط — وليس لا نهائيًا.

تكلفة بنية تحتية صفرية

يستخدم نفس مثيل Redis الذي يعمل أصلًا للتخزين المؤقت. مفاتيح حد المعدل ضئيلة الحجم (أقل من 100 بايت) وتنتهي تلقائيًا بعد 15 دقيقة.

5

الحماية من تعداد المستخدمين

تعداد المستخدمين هو القدرة على تحديد ما إذا كان بريد إلكتروني معيّن مسجّلًا في النظام بناءً على استجابة التطبيق. حتى فارق بسيط في رسائل الخطأ كافٍ لأداة آلية لبناء قائمة بالحسابات الصحيحة.

الثغرة السابقة

AuthService.php — قبل (معرّض للخطر) PHP 
// المسار 1 — بريد غير موجود أو كلمة مرور خاطئة → نفس الرسالة
if (! $user || ! Hash::check($password, $user->password)) {
    throw ValidationException::withMessages([
        'email' => ['The provided credentials are incorrect.'],
    ]);
}

// المسار 2 — حساب موقوف → رسالة مختلفة ← يكشف عن وجود المستخدم!
if (! $user->is_active) {
    throw ValidationException::withMessages([
        'email' => ['Your account has been deactivated.'],  // ← تعداد مستخدمين!
    ]);
}

كيف استغل المهاجمون هذه الثغرة

يرسل المهاجم POST /api/v1/auth/login ببريد إلكتروني مستهدف وكلمة مرور عشوائية:

  • "The provided credentials are incorrect." ← بريد غير موجود أو كلمة مرور خاطئة (غامض)
  • "Your account has been deactivated." ← البريد الإلكتروني موجود بالتأكيد في قاعدة البيانات

من خلال استهداف المستخدمين الموقوفين المعروفين، يمكن للمهاجم تأكيد أي رسائل بريد إلكتروني مسجّلة في TALENTO وبناء قائمة مستهدفة لهجمات لاحقة.

الإصلاح: استجابة موحدة

AuthService.php — بعد (محمي) PHP 
// سبب الفشل الداخلي (للتسجيل فقط — لا يُرسَل للعميل أبدًا)
$failureReason = match (true) {
    $user === null       => 'user_not_found',
    ! $passwordCorrect   => 'invalid_credentials',
    ! $user->is_active    => 'inactive_user',    // ← يُسجَّل السبب، لا يُكشَف
    default              => null,
};

if ($failureReason !== null) {
    RateLimiter::hit($throttleKey, self::DECAY_SECONDS);
    Log::warning('security.login_failed', ['reason' => $failureReason, ...]);

    // جميع مسارات الفشل تُعيد نفس الرسالة الخارجية بغض النظر عن السبب
    throw ValidationException::withMessages([
        'email' => ['The provided credentials are incorrect.'],
    ]);
}

أمثلة على الطلبات والاستجابات

السيناريو السبب الداخلي (مسجَّل) الاستجابة الخارجية HTTP
البريد الإلكتروني غير موجود في قاعدة البيانات user_not_found "The provided credentials are incorrect." 422

6

التخفيف من هجمات التوقيت

ما هو هجوم التوقيت؟

هجوم التوقيت يستغل الفروقات القابلة للقياس في المدة التي يستغرقها النظام للاستجابة لمدخلات مختلفة. في المصادقة: إذا استجاب الخادم بشكل أسرع لمحاولة تسجيل دخول ببريد غير موجود مقارنةً ببريد موجود (لكن كلمة مرور خاطئة)، يمكن للمهاجم قياس أوقات الاستجابة لتحديد أي عناوين البريد مسجّلة.

لماذا كان التطبيق القديم عُرضة للخطر؟

AuthService.php — قبل (معرّض) PHP 
$user = User::where('email', $email)->first();

// PHP Short-Circuit: إذا كان !$user صحيحًا، Hash::check لا يُستدعى أبدًا
if (! $user || ! Hash::check($password, $user->password)) {
    throw ...;
}

مسار البريد غير المعروف

  • بحث DB: ~2–5 مللي ثانية
  • !$user = true → Short-Circuit
  • Hash::check: يُتخطى تمامًا
  • الإجمالي: ~3–7 مللي ثانية

مسار البريد المعروف + كلمة مرور خاطئة

  • بحث DB: ~2–5 مللي ثانية
  • !$user = false → يكمل
  • Hash::check (bcrypt cost 12): ~150–250 مللي ثانية
  • الإجمالي: ~155–260 مللي ثانية

فارق ~150 مللي ثانية قابل للكشف إحصائيًا حتى مع وجود اهتزاز الشبكة. أدوات كـ timemap أو المتوسط الإحصائي عبر طلبات متعددة تُحدد هذا الفارق بموثوقية.

استراتيجية التجزئة الوهمية

AuthService.php — بعد (محمي) PHP 
// تجزئة bcrypt بتكلفة 12 صحيحة الصيغة — للتكافؤ الزمني فقط.
// الصيغة: $2y$12$ + 22 حرف ملح + 31 حرف تجزئة = 60 حرفًا
// Hash::check ينفذ الحساب الكامل ويُعيد false دائمًا.
private const DUMMY_HASH = '$2y$12$AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA';

$user            = User::where('email', $email)->first();
$passwordCorrect = Hash::check($password, $user?->password ?? self::DUMMY_HASH);
                                          ↑ bcrypt يعمل دائمًا بغض النظر
🔬
لماذا تعمل هذه التجزئة الوهمية؟ password_verify() في PHP (الدالة الداخلية التي تستخدمها Hash::check()) تستدعي crypt($password, $hash). لـ bcrypt، تستخرج crypt() الملح المكوّن من 22 حرفًا من سلسلة التجزئة وتشغّل خوارزمية bcrypt الكاملة بتكلفة 12 بصرف النظر عمّا إذا كانت التجزئة "حقيقية". النتيجة لن تساوي الجزء الوهمي من التجزئة، فتُعيد false — لكن الحساب الكامل (~150 مللي ثانية) يكون قد اكتمل.

❌ قبل: أوراكل التوقيت

  • بريد غير معروف: ~5 مللي ثانية (لا bcrypt)
  • كلمة مرور خاطئة: ~200 مللي ثانية (bcrypt يعمل)
  • الفارق: ~195 مللي ثانية — قابل للكشف
  • المهاجم يتعداد البريد عبر التوقيت

✅ بعد: التوقيت متكافئ

  • بريد غير معروف: ~200 مللي ثانية (bcrypt يعمل على الوهمية)
  • كلمة مرور خاطئة: ~200 مللي ثانية (bcrypt يعمل على الحقيقية)
  • الفارق: أقل من 5 مللي ثانية — ضجيج عادي
  • تحليل التوقيت لا يُعطي أي إشارة قابلة للاستغلال

انتهاء صلاحية رمز Sanctum

سلوك الرمز السابق

كان الإعداد 'expiration' => null في config/sanctum.php يعني أن Sanctum يُصدر رموزًا لا تنتهي صلاحيتها. بمجرد إصدار الرمز، يبقى صالحًا إلى الأبد ما لم يُحذف صراحةً عبر تسجيل الخروج. أي رمز مسروق يمنح وصولًا دائمًا للحساب.

تغيير الإعداد

❌ قبل

// config/sanctum.php
'expiration' => null,
// الرموز صالحة للأبد

✅ بعد

// config/sanctum.php
'expiration' => 10080,
// 7 أيام بالدقائق

كيف يعمل انتهاء صلاحية Sanctum

الآليةحارس Sanctum يفحص created_at + expiration_minutes > now() في كل طلب مصادق العمود المستخدمpersonal_access_tokens.created_at — موجود مسبقًا هجرة قاعدة بياناتغير مطلوبة — يستخدم العمود الموجود عند الانتهاءSanctum يحذف صف الرمز ويُعيد HTTP 401 Unauthenticated الرموز الحاليةالرموز الصادرة قبل هذا التغيير ستنتهي بعد 7 أيام من طابع created_at رموز الجلسةجلسات الـ SPA الخاصة بالطرف الأول غير متأثرة

التأثير على واجهة React الأمامية

يجب على واجهة React التعامل مع استجابات HTTP 401 من أي نقطة نهاية مصادقة:

  • استخدام Axios interceptor لالتقاط ردود 401 بشكل عام
  • عند 401: حذف الرمز المخزَّن وإعادة التوجيه إلى /login
  • هيكل استجابة تسجيل الدخول لم يتغير — لا تعديل مطلوب في كود المصادقة
  • الرموز تتجدد طبيعيًا: كل تسجيل دخول ناجح يُصدر رمزًا جديدًا لمدة 7 أيام

التأثير على تطبيق Flutter

يجب على تطبيق Flutter التعامل مع ردود 401 عبر Dio أو http client:

  • إضافة DioInterceptor لاعتراض ردود 401
  • عند 401: حذف الرمز من FlutterSecureStorage
  • التنقل بالمستخدم إلى شاشة تسجيل الدخول
  • صيغة Bearer token لم تتغير — لا تعديل في كود طلبات HTTP
⚠️
لا يوجد آلية Refresh Token: رموز Sanctum الشخصية لا تدعم تجديد الرمز. عند انتهاء الصلاحية، يجب على المستخدم تسجيل الدخول مجددًا. هذا مقصود أمنيًا — يحدّ من تأثير سرقة الرمز إلى 7 أيام كحد أقصى. إذا كانت جلسات أطول ضرورية، يمكن رفع expiration إلى 20160 (14 يومًا).

تكامل Redis

لم يُنشأ أي خدمة Redis جديدة أو غلاف أو إعداد. تمت إعادة استخدام مثيل Redis الحالي وبنية RedisCacheService القائمة. واجهة Laravel RateLimiter تستخدم CACHE_STORE=redis تلقائيًا.

كيف يستخدم RateLimiter نظام Redis

تعتمد واجهة Laravel RateLimiter على مخزن الكاش المعرَّف في CACHE_STORE. بما أن المشروع يضبط CACHE_STORE=redis، فإن RateLimiter::hit() يكتب مباشرةً إلى Redis مع TTL تلقائي. لا إعداد إضافي مطلوب.

.env — إعداد الكاش ENV 
CACHE_STORE=redis          # RateLimiter يستخدم هذا تلقائيًا
REDIS_CLIENT=predis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379

مفاتيح Redis التي ينشئها محدد المعدل

نمط المفتاح مثال TTL القيمة
Laravel-cache-login:{email}:{ip} Laravel-cache-login:user@ex.com:1.2.3.4 900 ثانية عدد صحيح (عدد المحاولات)
Laravel-cache-login:{email}:{ip}:timer Laravel-cache-login:user@ex.com:1.2.3.4:timer 900 ثانية Unix timestamp لأول محاولة

كل محاولة تسجيل دخول فاشلة وكل حدث حظر يُسجَّل الآن باستخدام Log::warning(). تذهب السجلات إلى قناة السجل المُعدَّة (LOG_CHANNEL=stack) وتُكتب في storage/logs/laravel.log.

الحدث اسم القناة المحفّز قيمة السبب

شرح التحسين

قبل التقوية، كانت الآلية الأمنية الوحيدة في نظام المصادقة هي تجزئة bcrypt بـ12 جولة — وهي خط أساسي يجب أن تمتلكه جميع الأنظمة. كل بُعد أمني آخر كان غائبًا تمامًا: لا تقييد معدل، لا تتبع محاولات، لا حماية من التعداد، لا دفاع زمني، ولا انتهاء صلاحية للرموز.

بعد التقوية، يُطبّق النظام المجموعة الكاملة من دفاعات المصادقة المعيارية المطلوبة لأي API في بيئة إنتاجية. النقطة المتبقية محجوبة لغياب تدفق إعادة تعيين كلمة المرور مع حمايته الخاصة — وهي ميزة غير مُنفَّذة حاليًا في المشروع.

التوصيات المتبقية

منخفض

إعداد APP_NAME

تحديث APP_NAME=Laravel إلى APP_NAME=Talento في ملف .env لضمان عزل بادئة مفاتيح Redis الصحيحة. البادئة الحالية: Laravel-cache-.

منخفض

تقييد إعادة تعيين كلمة المرور (عند التنفيذ)

المشروع لا يملك تدفق إعادة تعيين كلمة المرور. عند إضافته، يجب أن يتضمن محدد معدل خاصًا به لمنع إرسال بريد إعادة تعيين مفرط. إعداد framework يحتوي بالفعل على 'throttle' => 60 تحت وسيط كلمات المرور.

منخفض

إعداد ثقة الخادم الوكيل (TrustProxies)

إذا كان API يعمل خلف موازن تحميل، فقد يُعيد $request->ip() عنوان IP الخادم الوكيل بدلًا من العميل الحقيقي. قم بإعداد middleware TrustProxies مع نطاق IP الخادم الوكيل لضمان تقييد المعدل الصحيح لكل عميل.

ملاحظة

APP_DEBUG في بيئة الإنتاج

الإعداد APP_DEBUG=true مضبوط في .env الحالي. تأكد من ضبطه على false في بيئة الإنتاج لمنع ظهور تتبعات المكدس ومسارات الملفات الداخلية في ردود الأخطاء.
