مقدمة

🔴 ما هو Redis؟

Redis هو نظام تخزين بيانات مفتوح المصدر يعمل في الذاكرة العشوائية (RAM)، ويُعرَّف اختصاراً بـ Remote Dictionary Server. يتميّز بسرعته الفائقة في القراءة والكتابة مقارنةً بقواعد البيانات التقليدية، إذ يصل زمن الاستجابة عادةً إلى أقل من ميلي ثانية واحدة.

يدعم Redis هياكل بيانات متعددة: Strings، Hashes، Lists، Sets، Sorted Sets. في مشروع TALENTO يُستخدم حصراً كـ Cache Layer باستخدام هيكل Strings (JSON مُتسلسَل).

🤔 لماذا أضفنا Redis إلى TALENTO؟

بعد مرحلة التطوير الأولى، لاحظنا أن الطلبات المتكررة على قوائم المستخدمين والوظائف والدورات تُرهق قاعدة بيانات MySQL بشكل غير ضروري. القوائم المرقَّمة (Paginated Lists) تُنفَّذ عبر JOIN وشروط تصفية معقدة، وتعود بنفس النتيجة لعشرات الثواني.

الحل: تخزين نتيجة الاستعلام الأول في Redis، وإعادة استخدامها للطلبات التالية حتى تنتهي مدة الصلاحية أو تُبطَل يدوياً.

🗄 MySQL (قبل Redis)

  • ⏱ زمن استجابة: 50–300ms
  • 💾 كل طلب = استعلام SQL جديد
  • 🔄 عبء ثابت على قاعدة البيانات
  • 📊 JOIN معقد في كل مرة
  • 🐢 يتباطأ مع زيادة البيانات

⚡ Redis (بعد الإضافة)

  • ⏱ زمن استجابة: <1ms
  • 💾 طلبات متعددة = استجابة واحدة من الذاكرة
  • 🔄 MySQL يُستعلَم فقط عند انتهاء الكاش
  • 📊 لا JOIN — بيانات جاهزة مُتسلسَلة
  • 🚀 أداء ثابت بغض النظر عن حجم البيانات

معمارية Redis داخل TALENTO

🏗 نظرة عامة على المكونات

تتكون البنية من ثلاث طبقات: طبقة التطبيق (Laravel)، طبقة الكاش (Redis)، وطبقة البيانات الدائمة (MySQL). RedisCacheService هو الوسيط المركزي الوحيد بين التطبيق وRedis — لا يوجد أي تعامل مباشر مع Redis خارج هذه الخدمة.

📦 RedisCacheService — الوسيط المركزي

جميع عمليات Redis تمر عبر RedisCacheService المُسجَّل كـ Singleton في AppServiceProvider. هذا يضمن:

  • ✅ معالجة موحَّدة للأخطاء
  • ✅ تسجيل مركزي للمشاكل
  • ✅ لا يصل Redis Exception إلى المستخدم أبداً
  • ✅ سهولة الاختبار والاستبدال

⚙️ إعداد الاتصال

يستخدم المشروع مكتبة Predis (PHP Redis client). الإعداد في config/database.php:

config/database.php
'redis' => [
  'client' => env('REDIS_CLIENT', 'phpredis'),
  'options' => [
    'prefix' => env('REDIS_PREFIX',
      Str::slug(env('APP_NAME')).'-database-'
    ),
  ],
  'default' => ['database' => 0],
  'cache'   => ['database' => 1],
]

RedisCacheService — شرح تفصيلي

هذه الخدمة موجودة في app/Services/RedisCacheService.php وتُغلِّف جميع عمليات Redis في wrapper آمن يمتص الاستثناءات ويُسجِّلها دون أن تنتشر للمستخدم.

get() get(string $key): mixed

يجلب قيمة مخزَّنة في Redis ويُفكِّكها من JSON. يُعيد null إذا كان المفتاح غير موجود أو عند حدوث خطأ في Redis.

PHP
public function get(string $key): mixed
{
    try {
        $raw = Redis::get($key);
        return $raw !== null
            ? json_decode($raw, true)
            : null;
    } catch (\Throwable $e) {
        Log::warning("Redis get failed [{$key}]: {$e->getMessage()}");
        return null; // fallback to DB
    }
}
💡
سلوك الـ Fallback
عند إعادة null، تُنفِّذ الخدمة المُستدعِية استعلام قاعدة البيانات تلقائياً وتخزِّن النتيجة في Redis. هذا يعني أن فشل Redis لا يُعطِّل التطبيق.
set() set(string $key, mixed $value, int $ttl): void

يُخزِّن قيمة في Redis بعد تحويلها إلى JSON، مع تحديد مدة انتهاء الصلاحية بالثواني. يستخدم SETEX لضمان وجود TTL دائماً.

PHP
public function set(string $key, mixed $value, int $ttl): void
{
    try {
        Redis::setex($key, $ttl, json_encode($value));
    } catch (\Throwable $e) {
        Log::warning("Redis set failed [{$key}]: ...");
    }
}
remember() remember(string $key, int $ttl, callable $callback): mixed

يُنفِّذ نمط Cache-Aside كاملاً في خطوة واحدة: يتحقق من Redis، وإذا لم يجد البيانات ينفِّذ الـ $callback، يخزِّن النتيجة، ويُعيدها. هذه الطريقة موجودة ولكنها غير مستخدمة حالياً في الكود — يُستعاض عنها بـ get()+set() يدوياً.

PHP
public function remember(string $key, int $ttl, callable $callback): mixed
{
    $cached = $this->get($key);
    if ($cached !== null) { return $cached; }

    $value = $callback();
    $this->set($key, $value, $ttl);
    return $value;
}
del() del(string ...$keys): void

يحذف مفتاحاً واحداً أو أكثر بشكل مباشر. يُستخدَم لحذف مفاتيح محددة ومعروفة المسمى مثل admin:plans.

PHP
// مثال: حذف كاش الخطط
$this->cache->del('admin:plans');

// حذف متعدد
$this->cache->del('key1', 'key2', 'key3');
delPattern() delPattern(string $pattern): void

يحذف جميع المفاتيح المطابقة لنمط معين باستخدام SCAN وليس KEYS. هذا الفرق حيوي في الإنتاج: الـ KEYS * يُوقف Redis مؤقتاً بينما SCAN يعمل بشكل تكراري غير محجوب.

⚠️
معالجة البادئة (Prefix)
Predis يُضيف البادئة تلقائياً للمفاتيح الفردية لكن لا يُضيفها لنمط SCAN. الكود يعالج هذا بإضافة البادئة يدوياً لنمط SCAN ثم حذفها من نتائج SCAN قبل استدعاء DEL.
PHP
public function delPattern(string $pattern): void
{
    $prefix      = config('database.redis.options.prefix', '');
    $scanPattern = $prefix . $pattern;
    $cursor      = '0';

    do {
        [$cursor, $keys] = Redis::scan(
            $cursor, 'match', $scanPattern, 'count', 100
        );
        if (!empty($keys)) {
            // حذف البادئة قبل إرسالها لـ DEL
            $unprefixed = array_map(
                fn($k) => substr($k, strlen($prefix)),
                $keys
            );
            Redis::del(...$unprefixed);
        }
    } while ($cursor !== '0');
}
filterKey() filterKey(string $prefix, array $filters): string

يُنشئ مفتاح كاش حتمياً (Deterministic) من بادئة ومجموعة فلاتر. يُرتِّب مصفوفة الفلاتر أبجدياً قبل تحويلها إلى MD5 لضمان أن نفس الفلاتر تُنتج دائماً نفس المفتاح بغض النظر عن الترتيب.

PHP
// المدخلات
$prefix  = 'admin:users';
$filters = ['page'=>1, 'per_page'=>20, 'role'=>'admin'];

// الكود
ksort($filters);  // ترتيب أبجدي
$filters = array_filter($filters); // حذف القيم الفارغة
return $prefix . ':' . md5(json_encode($filters));

// النتيجة
// "admin:users:a3f5c8d2e1b4..."
✅
لماذا MD5؟
لأن الفلاتر قد تكون طويلة أو تحتوي أحرفاً خاصة. MD5 يُنتج دائماً مفتاحاً نظيفاً بطول ثابت (32 حرفاً) آمناً لاستخدامه كـ Redis key.

مفاتيح الكاش — جدول كامل

جميع المفاتيح التالية تُخزَّن فعلياً بصيغة: {REDIS_PREFIX}{key} — على سبيل المثال: talento-database-admin:users:abc123

المفتاح (Key) الغرض TTL يُنشئه يُبطله النوع
admin:users:{hash} قائمة المستخدمين للمدير مع فلاتر 120s AdminService::listUsers activate, deactivate, create, register, addMember, removeMember String JSON
admin:company_requests:{hash} طلبات تسجيل الشركات للمدير 120s AdminService::listRegistrationRequests approve, reject, submitRequest String JSON
admin:plans قائمة خطط الاشتراك 3600s AdminService::listPlans ⚠️ لا إبطال String JSON
admin:courses:{hash} قائمة الدورات للوحة التحكم 120s Admin\CourseController::index store, update, destroy String JSON
public:courses:{hash} قائمة الدورات العامة 3600s CourseController::index أي تعديل من المدير String JSON
public:job_posts:{hash} قائمة الوظائف العامة 300s JobPostController::index create, update, delete String JSON
company:job_posts:{hash} قائمة وظائف الشركة المحددة 120s CompanyService::listCompanyJobPosts create, update, delete job post String JSON
job_seeker:{id}:recommended_courses:{n} توصيات دورات مخصَّصة لكل باحث 300s CourseController::recommended syncSkills String JSON

استراتيجية الإبطال (Cache Invalidation)

📌 المبدأ العام

يُستخدم Pattern-based Invalidation: عند أي تغيير في البيانات، يُحذف كل مفتاح يبدأ بالبادئة المناسبة باستخدام delPattern(). هذا يضمن حذف جميع الصفحات والفلاتر دفعة واحدة دون الحاجة لتتبع المفاتيح الفردية.

⚡ DB 0 — كاش التطبيق (نشط)

هذه القاعدة هي المستخدمة فعلياً من قبل RedisCacheService. تحتوي على جميع مفاتيح الكاش المذكورة في هذا التوثيق.

.env
REDIS_DB=0  # default connection

المفاتيح المخزَّنة:

  • 🔑 admin:users:*
  • 🔑 admin:company_requests:*
  • 🔑 admin:plans
  • 🔑 admin:courses:*
  • 🔑 public:courses:*
  • 🔑 public:job_posts:*
  • 🔑 company:job_posts:*
  • 🔑 job_seeker:*:recommended_courses:*

🔒 DB 1 — Laravel Cache Facade (فارغ)

هذه القاعدة مُهيَّأة لاستخدام Laravel's Cache:: facade في حال استخدامه. حالياً لا يستخدم المشروع Cache::get() أو Cache::remember() في أي مكان — فقط RedisCacheService المباشر.

.env
REDIS_CACHE_DB=1  # cache connection
CACHE_STORE=redis  # مُعيَّن للـ facade
⚠️
نقطة انتباه
وجود قاعدتي بيانات مستقلتين لنفس الغرض قد يُربك المطورين. يُنصح توحيد النهج إما بالاعتماد الكامل على RedisCacheService أو استخدام Cache:: facade.

القيود الحالية المعروفة

⚠️ admin:plans — لا يوجد إبطال

كاش الخطط له TTL ساعة كاملة ولا يُبطَل عند تعديل أي خطة. لو عطَّل المدير خطة معينة، ستظل مرئية للمدير لمدة تصل إلى 60 دقيقة.

✅ الحل: إضافة Model Observer للـ Plan يستدعي del('admin:plans') عند أي تغيير

⚠️ company:job_posts — إبطال واسع النطاق

عند إنشاء وظيفة لشركة A، يُحذف كاش جميع الشركات بسبب delPattern('company:job_posts:*'). شركة B لم تغيِّر شيئاً لكن كاشها يُحذف بلا مبرر.

✅ الحل: تضمين company_id في نمط الإبطال: delPattern("company:{$id}:job_posts:*")

⚠️ الوظائف المنتهية الصلاحية في public:job_posts

الوظائف التي يمر عليها expires_at تختفي من الاستعلامات الجديدة (بسبب scopePublished) لكنها قد تبقى في الكاش حتى انتهاء الـ TTL (5 دقائق). طالب وظيفة قد يرى وظيفة منتهية لفترة قصيرة.

✅ الحل: جدولة مهمة كل دقيقة لحذف public:job_posts:* أو تقليل TTL إلى 60 ثانية

⚠️ كاش التوصيات لا يُبطَل عند إضافة دورات

عند إنشاء دورة جديدة تُغطي مهارة مطلوبة، لن يرى الباحثون التوصية الجديدة حتى انتهاء الـ TTL (5 دقائق). الكاش يُبطَل فقط عند تغيير مهارات الباحث.

✅ الحل: في store/update/destroy الدورات — إضافة: delPattern('job_seeker:*:recommended_courses:*')

⚠️ كلمة مرور Redis غير مُعيَّنة

في جميع البيئات الحالية REDIS_PASSWORD=null. هذا يعني أن أي عملية على نفس الخادم يمكنها قراءة وتعديل وحذف بيانات Redis بدون أي مصادقة. البيانات المخزَّنة تشمل معلومات تعريفية شخصية (PII).

✅ الحل: تعيين كلمة مرور قوية في .env الإنتاج: REDIS_PASSWORD=strong_password_here

أفضل الممارسات المُطبَّقة

✅ نمط Cache-Aside (Lazy Loading)

لا يُملَأ الكاش مسبقاً. يُملَأ فقط عند أول طلب (Cache Miss)، ثم يُستخدَم للطلبات التالية. هذا يتجنب تخزين بيانات غير مطلوبة ويبسِّط إدارة الكاش.

✅ مفاتيح حتمية (Deterministic Keys)

استخدام filterKey() يضمن أن نفس مجموعة الفلاتر تُنتج دائماً نفس المفتاح بغض النظر عن ترتيب المعاملات في الرابط. ?page=1&role=admin تساوي ?role=admin&page=1.

✅ SCAN بدلاً من KEYS

استخدام SCAN في delPattern() بدلاً من KEYS * يضمن عدم حجب Redis Server أثناء عملية البحث. KEYS * قد يُوقف Redis لثوانٍ على قواعد بيانات كبيرة.

✅ امتصاص الاستثناءات (Exception Swallowing)

جميع عمليات Redis مُغلَّفة بـ try-catch. إذا تعطَّل Redis لأي سبب، يستمر التطبيق في العمل عبر MySQL مع تسجيل التحذير في الـ logs. لا يصل استثناء Redis للمستخدم أبداً.

✅ الإبطال بعد اكتمال العملية

في جميع المواضع، يُستدعى delPattern() بعد نجاح العملية في قاعدة البيانات وليس قبلها. هذا يتجنب حالة Race Condition حيث يُحذَف الكاش قبل اكتمال الكتابة في MySQL.

✅ Singleton Pattern للخدمة

تُسجَّل RedisCacheService كـ Singleton في AppServiceProvider. هذا يعني نسخة واحدة للتطبيق كله، مما يوفِّر الذاكرة ويضمن سلوكاً متسقاً.

✅ فصل تخزين النماذج حسب نوع الـ Cast

النماذج التي تُعاد منها Paginator (AdminService, CompanyService) تُخزِّن بـ getAttributes() للحفاظ على القيم الخام. Controllers التي تُخزِّن payload JSON كاملاً لا تحتاج لهذا الاحتياط.

التحسينات المستقبلية المُقترَحة

🚀 تحويل Queue إلى Redis

حالياً يستخدم المشروع Database Queue Driver (يخزِّن الوظائف في جدول MySQL). في بيئة الإنتاج عالية الحركة، هذا يُرهق قاعدة البيانات. الحل: QUEUE_CONNECTION=redis — يستخدم BLPOP الفوري بدلاً من polling قاعدة البيانات.

🍪 نقل Sessions إلى Redis

حالياً SESSION_DRIVER=database يُضيف قراءة جدول sessions لكل طلب مُصادَق. التحويل إلى SESSION_DRIVER=redis يُلغي هذه القراءة ويُحسِّن أداء كل طلب.

🔔 تخزين إشعارات Supabase مؤقتاً

حالياً كل إنشاء إشعار هو استدعاء متزامن (Synchronous) لقاعدة بيانات Supabase. عند إرسال إشعارات لعدة مستخدمين في نفس الوقت (مثل withdrawApplication) يتأخر الطلب بشكل متناسب. الحل: تحويل الإشعارات إلى Queued Job.

🌡 Cache Warming

حالياً الكاش يُبنَى بالطلب الأول (Cold Start). عند إعادة تشغيل Redis أو حذف الكاش، أول مستخدم يتحمَّل تكلفة الاستعلام. يمكن إضافة Artisan command يُسخِّن الكاش بالبيانات الأكثر طلباً عند بدء التشغيل.

📊 Redis Cluster / Redis Sentinel

حالياً يعمل Redis على Node واحد — نقطة فشل واحدة. في الإنتاج الكامل يُنصح بـ Redis Sentinel للـ High Availability أو Redis Cluster للتوزيع الأفقي عند تجاوز حجم البيانات ذاكرة خادم واحد.

🔒 Rate Limiting بـ Redis

Laravel's Throttle Middleware يستخدم Redis تلقائياً عند CACHE_STORE=redis. إضافة Rate Limiting على نقاط نهاية Login و Register و Company Registration تحمي من Brute Force وSpam بدون جهد إضافي.

📝 الخلاصة التنفيذية

منصة TALENTO تعتمد على Redis كطبقة كاش بين Laravel وMySQL. الأساس التقني راسخ: RedisCacheService آمنة ومرنة، والمفاتيح منظَّمة ومتسقة، والإبطال يعمل في أغلب مسارات البيانات. المشاكل الموجودة إما منخفضة الخطورة (TTL قصير يحمي منها) أو موثَّقة مع خطة إصلاح. قبل دخول الإنتاج الكامل، الأولوية القصوى هي: تعيين كلمة مرور Redis، إصلاح admin:plans، وتحويل Queue إلى Redis.
