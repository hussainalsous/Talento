لنظام البريد الإلكتروني

تحليل معمّق كامل لتدفّق إرسال الرسائل، ونظام التحقق من البريد، وطابور المهام، والتخزين، وخوارزميات دورة الحياة، والإعدادات — مبني على فحص مباشر للشيفرة المصدرية.

Laravel 13 · PHP 8.3 قاعدة البيانات: MySQL الطابور: Database Driver البريد: Gmail SMTP التاريخ: 2026-06-13

الملخص التنفيذي

يعتمد نظام البريد في المنصة على مسارَين منفصلين للإرسال غير المتزامن (Queued): إشعار التحقق VerifyEmailNotification، ومهمة الإرسال العامة SendEmail. جميع الرسائل تُرسَل عبر Gmail SMTP، وتُخزَّن مهام الطابور في جدول jobs داخل MySQL وليس في Redis.

1 تدفّق إرسال البريد

كل موضع في الشيفرة يُطلق إرسال بريد إلكتروني، وما الذي يُشغّله، والصنف المسؤول، والبيانات المُمرّرة.

# الرسالة المُشغِّل (Trigger) صنف Mailable / Notification المسار يُسجَّل في email_logs؟
1 تحقق بريد الباحث عن عمل AuthService::registerJobSeeker() VerifyEmailNotification → VerifyEmailMailable Notification لا ✗
2 استلام طلب تسجيل شركة AuthService::submitCompanyRegistrationRequest() SystemNotificationEmail SendEmail Job نعم ✓
3 تحقق بريد طلب الشركة AuthService::submitCompanyRegistrationRequest() VerifyRegistrationEmailMailable SendEmail Job نعم ✓
4 الموافقة على تسجيل شركة AdminService::approveRegistrationRequest() SystemNotificationEmail SendEmail Job نعم ✓
5 تحقق بريد المالك بعد الموافقة (شرطي) AdminService::approveRegistrationRequest() VerifyEmailNotification → VerifyEmailMailable Notification لا ✗
6 رفض طلب تسجيل شركة AdminService::rejectRegistrationRequest() SystemNotificationEmail SendEmail Job نعم ✓
7 إعادة إرسال بريد التحقق EmailVerificationController::resend() VerifyEmailNotification → VerifyEmailMailable Notification لا ✗

البيانات المُمرّرة إلى كل قالب

VerifyEmailMailable

  • User $user — لاشتقاق الاسم الأول
  • string $verificationUrl — رابط موقَّع
  • int $expiresInMinutes = 60

VerifyRegistrationEmailMailable

  • string $firstName
  • string $companyName
  • string $verificationUrl — صلاحية 48 ساعة
  • int $expiresInHours = 48

SystemNotificationEmail

  • string $title — عنوان الرسالة
  • string $message — نص متعدد الأسطر (يُحوَّل عبر nl2br)

CompanyNotificationEmail شيفرة ميتة

  • مُعرَّف لكن لا يُستدعى في أي مكان داخل المشروع
  • بنيته مطابقة لـ SystemNotificationEmail

تنويه مهم: رسائل «الترحيب» و«تغيير كلمة المرور» و«تفعيل/تعطيل الحساب» وغيرها ليست بريدًا إلكترونيًا — بل إشعارات داخل التطبيق (In-App) تُخزَّن عبر SupabaseNotificationService في Supabase، وقد استُثنيت من جدول الرسائل أعلاه.

2 نظام بريد التحقق

آلية التحقق، توليد الرابط الموقَّع وانتهاء صلاحيته، المسارات والمتحكّمات، وما يحدث قبل/بعد التحقق.

الآلية الأساسية

يطبّق نموذج User واجهة MustVerifyEmail، لكنه يتجاوز السلوك الافتراضي عبر دالّة مخصّصة تُرسل إشعارًا مُصمَّمًا (Branded) ومُجدوَلًا (Queued): 

// app/Models/User.php
class User extends Authenticatable implements MustVerifyEmail
{
    public function sendEmailVerificationNotification(): void
    {
        $this->notify(new VerifyEmailNotification());
    }
}

توليد الرابط الموقَّع

// VerifyEmailNotification::toMail()
$expiresInMinutes = (int) Config::get(
   'auth.verification.expire', 60);

$url = URL::temporarySignedRoute(
   'api.v1.email.verify',
   Carbon::now()->addMinutes($expiresInMinutes),
   [
     'id'   => $notifiable->getKey(),
     'hash' => sha1($notifiable
              ->getEmailForVerification()),
   ]
);

توقيع HMAC-SHA256 باستخدام APP_KEY، ويتضمّن الرابط بارامترات signature وexpires.

مدد انتهاء الصلاحية

تحقق المستخدم60 دقيقة
المصدرEMAIL_VERIFY_EXPIRE
تحقق طلب الشركة48 ساعة
المصدرثابت في الكود

الرابط لا يحمل التوكن نفسه؛ بل يُعاد اشتقاق sha1(email) ومقارنته زمنيًّا ثابتًا عبر hash_equals().

المسارات (Routes)

// routes/api.php — Route::prefix('email')->name('email.')
GET   /api/v1/email/verify/{id}/{hash}              // signed, بدون مصادقة
GET   /api/v1/email/verify-registration/{id}/{hash} // signed, بدون مصادقة
POST  /api/v1/email/verification-notification        // auth:sanctum + throttle:6,1

خطوات التحقّق داخل المتحكّم

1. hasValidSignature() — يرفض إن كان التوقيع غير صالح أو منتهيًا. عند الفشل يُميّز «منتهٍ» عن «غير صالح» بقراءة expires ومقارنته بـ time().
2. User::find($id) — يُرجع 404 إن لم يوجد.
3. hash_equals($hash, sha1($user->getEmailForVerification())) — مقارنة ثابتة زمنيًّا؛ 403 عند الفشل.
4. hasVerifiedEmail() — إن كان مُتحقّقًا مسبقًا يُعيد رسالة «مُتحقّق سلفًا» دون تغيير.
5. markEmailAsVerified() — يثبّت email_verified_at = now().

قبل التحقق وبعده: الحماية (Middleware)

البوابة verified

EnsureEmailIsVerified تُعيد:

{ "success": false,
  "message": "...",
  "email_verified": false }  // 403

المسارات المحميّة فقط

  • cvs.store — رفع السيرة الذاتية

مسارات الشركات غير محميّة بهذه البوابة عمدًا — لأن موافقة المشرف هي بوابة التحقق البديلة.

3 الطابور و Redis

أيّ الرسائل مُجدوَلة، إعدادات الاتصال، أسماء الطوابير، منطق إعادة المحاولة، والمهام الفاشلة.

تصحيح جوهري: برغم وجود Redis في المشروع، فإنّ محرّك الطابور هو database وليس Redis (QUEUE_CONNECTION=database). يُستخدم Redis حصرًا لطبقة التخزين المؤقّت (Cache) (CACHE_STORE=redis عبر عميل predis). لذا تُخزَّن مهام البريد المُجدوَلة في جدول jobs داخل MySQL، ولا تمرّ عبر Redis إطلاقًا.

مُجدوَل مقابل متزامن

البيئةالسلوك
الإنتاجdatabase (مُجدوَل)
الاختباراتsync (فوري)

جميع رسائل البريد مُجدوَلة في الإنتاج: المساران SendEmail وVerifyEmailNotification يطبّقان ShouldQueue.

منطق إعادة المحاولة

$tries3
$timeout30s
$backoff60s
retry_after90s

القيم متطابقة في كلا الصنفين. بعد 3 محاولات فاشلة تنتقل المهمة إلى failed_jobs.

إعداد اتصال الطابور

// config/queue.php
'default' => env('QUEUE_CONNECTION', 'database'),   // .env => database

'database' => [
    'driver'       => 'database',
    'table'        => 'jobs',
    'queue'        => 'default',
    'retry_after'  => 90,
    'after_commit' => true,   // لا يُجدوَل إلا بعد إتمام الـ transaction
],

'failed' => [
    'driver'   => 'database-uuids',
    'table'    => 'failed_jobs',
],

بنية حمولة المهمة المُسلسَلة (Payload)

يُسلسِل Laravel المهمة إلى JSON ويخزّنها في عمود payload بجدول jobs. يحوي الكائن المُسلسَل خصائص SendEmail العامّة (المُستلِم + الـ Mailable + سجلّ الـ log)، إضافةً إلى emailLogId الخاص الثابت:

{
  "uuid": "…",
  "displayName": "App\\Jobs\\SendEmail",
  "job": "Illuminate\\Queue\\CallQueuedHandler@call",
  "maxTries": 3, "timeout": 30, "backoff": 60,
  "data": {
    "commandName": "App\\Jobs\\SendEmail",
    "command": "O:18:\"App\\Jobs\\SendEmail\":… (serialized object)"
  }
}

بفضل SerializesModels، تُخزَّن النماذج كمُعرّفات (IDs) وتُسترجَع من قاعدة البيانات عند التنفيذ — وليس ككائنات كاملة.

jobs

المهام المُعلّقة

job_batches

دفعات المهام (غير مُستخدَم للبريد)

failed_jobs

المهام الفاشلة (UUID + exception)

Horizon: غير مُثبَّت (config/horizon.php غير موجود). وهو أصلًا يعمل فقط مع محرّك طابور Redis، بينما المشروع يستخدم محرّك database. إدارة العمّال تتمّ عبر php artisan queue:work التقليدي.

4 الطوابع الزمنية والتخزين

أين تُخزَّن الطوابع الزمنية المتعلّقة بالبريد، والجداول والأعمدة المعنيّة، وتسجيل أحداث الإرسال.

الجدول العمود المعنى
usersemail_verified_atطابع تحقّق المستخدم (nullable)
company_registration_requestsemail_verified_atتحقّق مُسبق للطلب — يُنقَل للمستخدم عند الموافقة (أُضيف 2026-06-13)
email_logscreated_at / updated_atوقت التسجيل ووقت تحديث الحالة (بديل sent_at)
email_logsstatusenum: pending / sent / failed
email_logserror_messageرسالة الاستثناء عند الفشل
failed_jobsfailed_at / exceptionطابع الفشل النهائي ونصّ الاستثناء

مخطّط جدول email_logs

// 2026_05_23_130716_create_email_logs_table.php
Schema::create('email_logs', function (Blueprint $table) {
    $table->uuid('id')->primary();
    $table->unsignedBigInteger('sender_id');  // أصبح nullable لاحقًا (2026-06-08)
    $table->string('subject');
    $table->string('to_email');
    $table->text('body');
    $table->enum('status', ['sent','failed','pending'])->default('pending');
    $table->text('error_message')->nullable();
    $table->timestamps();
    $table->foreign('sender_id')->references('id')->on('users')->onDelete('cascade');
});

فجوة في الرصد: جدول email_logs يُكتب فقط من مسار SendEmail. أما رسائل التحقّق المُرسَلة عبر مسار VerifyEmailNotification (رسائل ١، ٥، ٧) فلا تُسجَّل فيه. كما لا يوجد تتبّع لـ «تمّ التسليم / ارتداد / فُتحت» لأن إرسال Gmail SMTP لا يوفّر Webhooks.

5 خوارزمية ودورة الحياة الكاملة

تتبّع خطوة بخطوة لكامل دورة الحياة من التشغيل حتى التسليم، والمنطق الشرطي (عدم التكرار، التقييد، إلخ).

المسار (أ): تحقّق الباحث عن عمل عبر Notification

تسجيل عبر POST /api/v1/auth/job-seeker/register
registerJobSeeker() ينشئ User + JobSeeker داخل DB::transaction ويولّد توكن.
بعد الـ transaction: sendEmailVerificationNotification() (لضمان تثبيت صفّ المستخدم أولًا).
VerifyEmailNotification (ShouldQueue) تُسلسَل إلى جدول jobs.
العامل (queue:work) يلتقطها → toMail() يبني الرابط الموقَّع وVerifyEmailMailable.
يُرسَل عبر Gmail SMTP. النقر على الرابط → markEmailAsVerified() → إعادة توجيه إلى /email-verified.

المسار (ب): رسائل النظام عبر SendEmail Job

SendEmail::dispatch(recipient, mailable, log) — يُولَّد emailLogId (UUID) في الباني.
تُسلسَل المهمة إلى جدول jobs (نفس الـ UUID يُحفَظ داخل الحمولة).
handle(): EmailLog::firstOrCreate(['id'=>$emailLogId], …) — يمنع التكرار عند إعادة المحاولة.
Mail::to($recipient)->send($mailable) داخل try/catch.
نجاح → status = sent. فشل → status = failed + error_message ثم throw لتفعيل إعادة المحاولة.

المنطق الشرطي والحمايات

عدم الإرسال إن كان مُتحقّقًا

كلٌّ من resend() وverify() يتوقّف مبكّرًا إذا كان البريد مُتحقّقًا سلفًا.

تقييد المعدّل (Throttling)

throttle:6,1 — ٦ طلبات إعادة إرسال كحدّ أقصى في الدقيقة لكل مستخدم.

منع تكرار السجلّ (Idempotency)

emailLogId ثابت عبر المحاولات + firstOrCreate ⇒ صفّ واحد لكل إرسال منطقي.

نقل التحقّق المُسبق

عند الموافقة: لا يُرسَل بريد تحقّق إلا إذا لم يكن الطلب مُتحقّقًا سلفًا ($alreadyVerified).

التكامل الخارجي

لا يوجد تكامل مع خدمة بريد عبر API (مثل SES أو Mailgun أو Postmark). الإرسال يتمّ مباشرةً عبر Gmail SMTP باستخدام كلمة مرور تطبيق (App Password). قوالب ses/postmark/resend موجودة في config/mail.php الافتراضي لكنها غير مُفعَّلة.

7 الملاحظات والمخاطر والتوصيات

نتائج التدقيق التي تستحق الانتباه — مرتّبة حسب الأهمية. لم تُجرَ أي تعديلات على الشيفرة.

أمني

بيانات اعتماد Gmail SMTP في ملف .env

يحتوي .env على MAIL_PASSWORD (كلمة مرور تطبيق Gmail) وREDIS_PASSWORD. التوصية: التأكّد من أنّ .env ضمن .gitignore وعدم رفعه إطلاقًا، وتدوير كلمة المرور إن سبق كشفها.

رصد

رسائل التحقّق لا تُسجَّل في email_logs

مسار VerifyEmailNotification يتجاوز SendEmail ولا يكتب أي سجلّ. بالتالي رسائل التحقّق (للباحث، وللمالك بعد الموافقة، وإعادة الإرسال) لا أثر لها في جدول السجلّات. التوصية: توحيد التسجيل أو إضافة قناة رصد لرسائل الإشعارات.

نظافة

شيفرة ميتة: CompanyNotificationEmail

الصنف مُعرَّف بالكامل لكنه لا يُستدعى في أي موضع. بنيته نسخة طبق الأصل من SystemNotificationEmail. التوصية: حذفه أو استخدامه فعليًّا لرسائل الشركات لتجنّب الالتباس.

معلومة

لا يوجد تدفّق «استعادة كلمة المرور» عبر البريد

المسار PATCH /auth/password هو تغيير كلمة مرور لمستخدم مُصادَق فقط، يُنتج إشعارًا داخل التطبيق دون أي بريد. لا توجد آلية «نسيت كلمة المرور / إعادة تعيين عبر رابط». التوصية: إضافتها إن كانت ضمن المتطلّبات.

إيجابي

ممارسات سليمة مرصودة

  • تنقية مُدخلات HTML في كل القوالب عبر htmlspecialchars().
  • مقارنة الهاش الثابتة زمنيًّا عبر hash_equals() (مقاومة لهجمات التوقيت).
  • إرسال غير متزامن بالكامل + إعادة محاولة + idempotency على السجلّ.
  • عدم تمرير البارامترات الحسّاسة (signature/expires/hash) إلى الواجهة عند إعادة التوجيه.
  • استدعاء إرسال التحقّق بعد الـ transaction + خيار after_commit.
